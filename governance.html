06 — Operating

Governance

A brand system only earns its keep if it actually gets used day to day, not filed away after a kickoff call. This page covers how the system stays alive, how files stay organized, and what happens when a new situation isn't covered anywhere else in this document.

New — not previously defined

This is a living document

This site, not a static PDF, is the source of truth going forward — meaning updates happen by editing these pages directly rather than producing a "v2" document that quietly diverges from what's actually in use. Anyone touching a thumbnail, social post, or app screen should be able to open this and check their work against it.

File & naming conventions

As the content catalogue grows, assets need a naming pattern that stays sortable and searchable without anyone having to remember a special case.

faithstream_thumbnail_abejoye_1920x1080_v2.png

Pattern: brand_assettype_title-or-context_dimensions_version.ext — lowercase, hyphens within segments, underscores between them. Applies to thumbnails, key art, social graphics, and exported UI assets alike.

Approval path for new applications

When a new situation comes up that this document doesn't explicitly cover — a partnership co-brand, a new content vertical, a markedly different UI pattern — this is the path rather than making the call solo under deadline pressure.

1

Check this document first

Most genuinely new-feeling situations are actually covered by an existing rule once you look — check Foundations for the underlying principle before assuming nothing applies.

2

If it's genuinely uncovered, propose within the system

Draft the new application using the existing tokens (color, type, spacing) before introducing anything new. Most "new" needs turn out to be a new combination of existing pieces, not a reason to add a new color or font.

3

Flag genuinely new additions for review

If a new token (a color, a type weight, a component pattern) seems truly necessary, flag it the same way the open questions in this document are flagged — visibly, with the reasoning attached — rather than slipping it in silently.

4

Update this document

Once a new pattern is approved, it gets added here so the next person doesn't have to rediscover or re-argue it.

Sub-brand framework (future)

House of Faith's other initiatives — the publishing arm, live events, regional production hubs — may eventually need their own distinct identity within the master brand, similar to how a university's individual schools operate under one parent brand. Not needed now, but worth establishing the principle early: any sub-brand keeps the core purple as its anchor and borrows the master type system, but may develop its own accent tint and motif within that frame, the same way the genre color-coding system on the Motifs page works.

Why this matters now
Right now, work on thumbnails and platform look-and-feel happens without this foundation in place — every new asset ends up making its own typography and color decisions because there's nothing to check against. With this system in place, every future decision, from a poster to an app screen, gets faster and more consistent because there's a clear standard to check against instead of starting from scratch each time. Nothing here changes the logo or the core purple identity already shipped — it builds the missing structure around them.
