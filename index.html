FaithStream / House of Faith — Complete Brand System

A build-up, not a rebrand.

This is the full reference for how the FaithStream brand looks, sounds, and behaves — across marketing, the web platform, and the mobile app. Every section is grounded in what's already shipped: the existing logo, the existing purple palette, the existing Inter/Syne type pairing already running in production. This document fills in the rules, range, and detail around what's there — it does not replace it.

Version 1.0 Status Draft for review

Why the one-page guide reached its limit

FaithStream launched with a one-page mini brand guide — the right tool for that stage. It established the logo, named four colors, pointed to a typeface, and got a product shipped. That job is done. What a one-page guide cannot do is serve a growing product, a production team creating thumbnails weekly, a developer building new screens, and a marketing team running campaigns — all at the same time, all consistently. The gaps it left behind showed up in the live product before this system was started.

The mini guide — what it covered
○
Logo — one contextPrimary logo on dark. No icon-only rules, no light-mode variant, no minimum size, no co-brand guidance.
○
4 color tokensNamed plum, magenta, lavender, cream. No tints, no shades, no usage rules — nothing to prevent an unrelated color from slipping in.
○
Typography — named, not ruledMiller Text named for the wordmark. No system for what to use in headlines, UI, or body copy. No scale.
○
Photography — "TBC"Direction explicitly marked as to-be-confirmed. Left open.
○
No misuse rulesDescribed in prose only. No visual gallery, no examples of what wrong looks like.
○
No motifs or patternsNo graphic language beyond the logo. Nothing to fill a background, a banner, or a loading state.
○
No UI or component rulesNothing about how the brand behaves inside a button, a card, a modal, or a mobile tab bar.
○
No governanceNo process for what happens when a new situation isn't covered, or when the guide itself needs updating.

What the gaps produced in the live product

These aren't hypothetical risks. Each finding below was found in the FaithStream product code before this system was started — direct evidence of what happens when a team builds fast without a documented standard to pull from.

01
Unrelated colors entered the codebase
The browse panel's five genre tiles used gradients spanning blue, green, and brown — hue families with no relationship to the purple brand. Without a documented color rule, each gradient was invented individually. The new system replaces all five with purple-family gradients.
Found in live code — .bg-thumb.g1–g5
02
A third typeface appeared with no decision behind it
One hero slide used Cinzel — a serif display face that appears nowhere in the brand guide and nowhere else in the product. It arrived for a single moment of flourish, and was never removed. The new system flags it for retirement and formalizes the two-family rule: Syne + Inter, everywhere, no exceptions.
Found in live code — .hero-tagline
03
A Netflix convention color was adopted without a brand decision
Green (#4ade80) was used for "Match %" indicators in hover previews and modals — imported from Netflix's visual language without a deliberate choice about whether FaithStream should use it. The new system surfaces this as an open decision to resolve, rather than letting it silently spread.
Found in live code — modal match % display
04
Icon names broke across a Lucide version update
Three nav drawer icons (Series, Worship, Podcasts) used tv-2, music-4, and mic-2 — names that Lucide has since renamed. Without documented icon names, the breakage went unnoticed. The new system documents every icon name actually in use.
Found in live code — nav drawer icon names
05
Logo export inconsistency across icon variants
The purple icon file includes the full circular badge, but the white and black icon files are the bare flame mark only — no badge. The result is three "icon" files that aren't interchangeable. The new system flags this for a re-export and documents exactly which file to use in each context until that fix lands.
Found in /assets/logo/ — icon variant exports
06
No misuse rules meant no shared definition of "wrong"
The mini guide described logo rules in prose. Prose rules don't stop anyone from rotating the logo 20 degrees on a banner, placing it on a low-contrast magenta background, or stretching it to fill a space. The new system has a visual gallery of exactly 20+ documented violations — using the real logo files — so there is no ambiguity about what wrong looks like.
Gap in mini guide — no misuse gallery
Why this should be the standard going forward
A one-page guide is a starting point, not an operating document. FaithStream now has a production team shipping thumbnails weekly, developers building new screens, and marketing creating campaign assets — all working in parallel. At that scale, a guide that lives in one PDF and covers four colors and a logo is not a brand system: it is a note. This document is the system. It is grounded in what has already shipped, it documents the decisions the team has already made implicitly, and it gives every role — design, development, content, marketing — a single place to pull from rather than improvising per project. Adopting it as the standard does not require anyone to rebuild anything. It requires the team to stop building around it and start building from it.

Three operating principles

Everything in this system traces back to three rules. When a new design decision isn't explicitly covered anywhere else in this document, check it against these first.

1

Purple carries the brand

The purple family is the single thing that makes FaithStream recognizable at a glance. Every accent, tint, and shade in this system stays inside that family — nothing borrowed from outside it competes for that job.

2

Confident, not loud

FaithStream competes with Netflix-caliber platforms, but the tone is warm and dignified, not aggressive. Typography, color, and copy should read as premium and considered — never as shouting for attention.

3

One system, every surface

A thumbnail, a social post, an app screen, and a donation page should all feel like the same product made by the same team — because they're built from the same small set of tokens, not improvised per project.

Where this comes from
This system is grounded directly in the live FaithStream product code — the same CSS variables, component patterns, and interaction states already shipped are referenced and reconciled throughout, rather than invented from scratch. Where something in the current build is inconsistent (a stray color, an unplanned font), it's flagged explicitly rather than silently fixed, so the decision is visible and deliberate.
